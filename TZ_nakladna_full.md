# ТЗ: Система генерації накладної (вимоги) — Додаток 25

## Призначення

Веб-функціонал для створення/редагування військової накладної (вимоги) з
наступним експортом у заповнений Excel-шаблон. Стек бекенда — Python
(фреймворк і БД — на розсуд розробника; рекомендація нижче).

Ланцюг: **веб-форма → збереження запису в БД (зі снапшотом даних) →
експорт у Excel-шаблон `nakladna1_clean.xlsx`**.

---

## 1. Архітектурні принципи (зафіксовані рішення)

1. **Снапшот-документ.** Накладна при створенні фіксує копії всіх значень
   (назви майна, ціни, ПІБ, посади, звання, підрозділи). Подальші зміни
   довідників НЕ змінюють раніше створені накладні. Документ не змінюється
   заднім числом.
2. **Редагування** існуючої накладної бере дані з БД (зі збереженого
   снапшоту), а не з поточних довідників.
3. **Шаблон Excel уже очищений** від блоку фінансово-економічного органу
   (див. розділ 5). Модуль НЕ робить цього видалення — працює з готовим
   `nakladna1_clean.xlsx`.
4. **Динаміка рядків майна** — залишається в модулі експорту: кількість
   рядків позицій змінна (1..N), модуль додає/видаляє рядки при кожному
   експорті (див. розділ 6).

### Рекомендація стеку (необов'язкова)

Django + PostgreSQL: ORM-моделі для довідників, вбудована адмінка для їх
ведення, форми для накладної. Якщо обрано Flask/FastAPI — SQLAlchemy +
окремий CRUD. ТЗ написано стек-нейтрально; усі вимоги застосовні до
будь-якого варіанта.

## 2. Джерела даних

| Код | Джерело | Природа |
|-----|---------|---------|
| `ORG` | Налаштування організації | Єдині на все, рідко змінюються |
| `SETT-SERV` | Налаштування → довідник «Служби» | Список |
| `SETT-OPTYPE` | Налаштування → довідник «Типи операцій» | Список |
| `PERS` | Довідник «Особи» | Список, ключ — стабільний id рядка |
| `NOM` | Довідник «Номенклатура» (майно) | Список |
| `OP` | Дані конкретної операції | Вводяться щоразу |
| `CALC` | Обчислюється модулем | — |

### 2.1 Структури довідників

**ORG — Налаштування організації** (одна група полів):
- `unit_name` — Найменування (повне офіційне)
- `edrpou` — Код ЄДРПОУ
- `location` — Місце дислокації

**SETT-SERV — довідник «Служби»**, рядок:
- `service_name` — назва служби
- `chief_post` — посада начальника служби
- `chief_name` — ПІБ начальника служби

**SETT-OPTYPE — довідник «Типи операцій»**, рядок:
- `op_type_name` — назва типу операції
- `number_prefix` — префікс номера накладної (редагований)

**PERS — довідник «Особи»**, рядок (стабільний `id`):
- `subdivision` — підрозділ (унікальний у межах довідника)
- `full_name` — ПІБ
- `post` — посада
- `rank` — звання
- Один рядок на підрозділ. Звання зберігається ОКРЕМИМ полем (бо в одному
  місці документа потрібно «звання + ПІБ», а в іншому — лише ПІБ).

**NOM — довідник «Номенклатура»**, рядок:
- `name` — назва майна
- `nomen_code` — код номенклатури
- `unit` — одиниця виміру
- `category` — категорія (сорт)
- `price` — вартість за одиницю

## 3. Веб-форма накладної — поля

Поля згруповані за блоками. Тип: `auto` — підтягується автоматично/тільки
для читання; `select` — вибір зі списку; `input` — ручне введення;
`calc` — обчислюється.

### 3.1 Реквізити документа

| Поле форми | Тип | Джерело | Логіка |
|-----------|-----|---------|--------|
| Найменування організації | auto | `ORG.unit_name` | read-only |
| ЄДРПОУ | auto | `ORG.edrpou` | read-only |
| Місце складання | auto | `ORG.location` | read-only |
| Тип операції | select | `SETT-OPTYPE` | вибір рядка |
| Номер накладної | input+auto | `SETT-OPTYPE.number_prefix` + автономер | див. 3.1.1 |
| Дата складання | input | `OP` | default = сьогодні (при створенні); при редагуванні — з БД |
| Дата операції | input | `OP` | default = сьогодні (при створенні); при редагуванні — з БД |
| «Дійсна до» | calc | `CALC` | = дата складання + 3 дні |
| Служба | select | `SETT-SERV` | вибір рядка; тягне назву + посаду + ПІБ начальника |
| Підстава (мета) | input | `OP` | текстове поле |
| Вид операції | (= Тип операції) | `SETT-OPTYPE.op_type_name` | назва обраного типу |

#### 3.1.1 Логіка номера накладної

- Кожен **Тип операції** має власний `number_prefix` (редагований у
  налаштуваннях типу).
- Підсумковий номер = `<number_prefix>` + `<порядковий_номер>`.
- Порядковий номер — автоінкремент (у прикладі `85/635-66` числова частина,
  що інкрементується, = `66`; префікс = `85/635-`).
- Поле редаговане користувачем вручну.
- При збереженні — перевірка унікальності номера; якщо такий номер уже існує
  → **попередження про дублікат** (не блокування, рішення за користувачем).
- Логіку скидання нумерації (напр. щороку) — поки не реалізовувати,
  залишити точку розширення (TODO).

### 3.2 Сторона «передає / приймає»

| Поле форми | Тип | Джерело | Логіка |
|-----------|-----|---------|--------|
| Підрозділ-відправник | select | `PERS` (список підрозділів) | вибір підрозділу |
| → Посада (здав) | auto | `PERS.post` обраного рядка | автозаповнення |
| → ПІБ (здав) | auto | `PERS.full_name` обраного рядка | автозаповнення |
| Підрозділ-отримувач | select | `PERS` (список підрозділів) | вибір підрозділу |
| → Звання+ПІБ (одержувач) | auto | `PERS.rank`+" "+`PERS.full_name` | автозаповнення |
| → Посада (прийняв) | auto | `PERS.post` обраного рядка | автозаповнення |
| → ПІБ (прийняв) | auto | `PERS.full_name` обраного рядка | автозаповнення |

Примітка: одержувач і «прийняв» — одна й та сама особа (підрозділ-отримувач).
ПІБ цієї особи виводиться у двох місцях документа в різних формах:
зі званням (одержувач) і без звання (підпис «прийняв»).

### 3.3 Підпис фінансової служби

| Поле форми | Тип | Джерело | Логіка |
|-----------|-----|---------|--------|
| Посадова особа фінслужби | select | `PERS` (вибір **за посадою**) | вибір рядка довідника Особи зі списку посад |
| → Посада | auto | `PERS.post` обраного рядка | |
| → ПІБ | auto | `PERS.full_name` обраного рядка | |

Це той самий довідник `PERS`. Відмінність від 3.2: тут користувач вибирає
рядок **за посадою** (не за підрозділом), бо поля підрозділу для фінслужби
немає. Адресація рядка — за стабільним `id`, посада/підрозділ — атрибути.

### 3.4 Позиції майна (динамічна таблиця)

Користувач додає N рядків. Кожен рядок:

| Поле форми | Тип | Джерело |
|-----------|-----|---------|
| Майно | select | `NOM` (вибір рядка) |
| → Код номенклатури | auto (снапшот) | `NOM.nomen_code` |
| → Одиниця виміру | auto (снапшот) | `NOM.unit` |
| → Категорія (сорт) | auto (снапшот) | `NOM.category` |
| → Вартість за одиницю | auto (снапшот) | `NOM.price` |
| Кількість відправлено | input | `OP` |
| Кількість прийнято | input | `OP` |
| Примітка | input | `OP` |

Усі `auto`-поля фіксуються снапшотом у момент створення накладної (зміна
довідника `NOM` пізніше не змінює документ — підтверджено: «ціна не
зміниться»).

## 4. Схема БД (логічна)

Довідники: `Organization` (ORG, singleton/одна група), `Service`
(SETT-SERV), `OperationType` (SETT-OPTYPE), `Person` (PERS),
`Nomenclature` (NOM).

**Документ накладної — `Waybill`:**
- Реквізити: `number`, `date_compiled`, `date_operation`, `basis`,
  `internal_note` тощо.
- FK на `OperationType`, `Service`, на `Person` (відправник, отримувач,
  фінпідписант) — для аналітики/посилань.
- **Снапшот-поля** (копії на момент створення, саме вони рендеряться в
  Excel): `snap_unit_name`, `snap_edrpou`, `snap_location`,
  `snap_service_name`, `snap_service_chief_post`, `snap_service_chief_name`,
  `snap_op_type_name`, `snap_sender_subdiv`, `snap_sender_post`,
  `snap_sender_name`, `snap_recv_subdiv`, `snap_recv_rank`,
  `snap_recv_name`, `snap_recv_post`, `snap_fin_post`, `snap_fin_name`.

**Позиції — `WaybillItem`** (FK → `Waybill`, порядок `position_no`):
- FK на `Nomenclature` (посилання) +
- Снапшот: `snap_name`, `snap_nomen_code`, `snap_unit`, `snap_category`,
  `snap_price`, `qty_sent`, `qty_received`, `note`.

> Принцип: Excel-експорт читає ТІЛЬКИ снапшот-поля, ніколи не звертається
> до поточних довідників. FK лишаються для звітності/фільтрів.

## 6. Динаміка рядків позицій (логіка модуля експорту)

Шаблон містить 5 слотів (рядки 20–24), підсумок — рядок 25.

- **N = 5** — заповнити 20–24.
- **N < 5** — заповнити 20..(19+N); рядки (20+N)..24 видалити
  `delete_rows`. Перевірити коректність merge/формул підсумку після зсуву.
- **N > 5** — `insert_rows(25, N-5)`; стиль кожного нового рядка скопіювати
  з рядка-зразка 20 поклітинно (`copy.copy(src._style)`) — openpyxl при
  insert НЕ копіює стилі. Зберегти особливості рамок: колонка A —
  `left=None`, колонка C — `bottom=None`.

Після зміни кількості рядків:
- формули підсумку: `SUM(...20:{last})`;
- перебудувати/перевірити merge у блоках нижче (`A{tot}:F{tot}`,
  `A{...}:J{...}` пропису, `A{fin}:C{fin+2}` фінпідпису);
- оновити `print_area` = `A1:J{new_last}`;
- адресувати блоки за «якорями», не за константними номерами.

## 7. Обчислювані значення

1. `position_no` = 1..N.
2. Колонка I = формула `=F{r}*G{r}` (НЕ рахувати в Python).
3. Підсумки = формули `SUM`.
4. `B6` «Дійсна до» = дата складання + 3 дні, формат
   `Дійсна до "DD" <місяць у родовому відмінку> YYYY року`
   (напр. «травня»).
5. **Кількість прописом** (`C29`) = сума `qty_sent` усіх позицій,
   українською прописом, узгоджена зі словом «одиниць» (стоїть константою
   в `J29`). Напр. 7 → «сім».
6. **Сума прописом** (`A31`) = `SUM(snap_price*qty_sent)` українською:
   `на суму <гривні прописом> <грн-слово відмінок> <копійки цифри 2 знаки> <коп-слово відмінок>`.
   Приклад для 76453.20 → `на суму сімдесят шість тисяч чотириста
   п'ятдесят три гривні 20 копійок`.
   Узгодження: «гривня/гривні/гривень», «копійка/копійки/копійок».
   Гроші рахувати на `Decimal`, не float.

> ⚠️ В оригінальному xlsx пропис був зіпсований (латинська `i` всередині
> кирилиці: `сiм`, `гривнi`). НЕ копіювати. Реалізувати коректний
> український модуль числівників (рекомендація: `num2words` lang='uk' з
> кастомним узгодженням валюти, або власний модуль). Покрити тестами:
> 0,1,2,5,7,11,21,100,1000, копійки 00/01/20.

## 8. Тести (мінімум)

1. N=5 → структура як еталон; I20..I24 = price*qty; I25 = сума.
2. N=1 → зайві слоти видалені, підсумок/підписи зсунуті, merge цілі,
   print_area оновлено.
3. N=12 → 7 рядків вставлено, стилі/рамки скопійовані (A.left=None,
   C.bottom=None збережені), формули SUM коректні.
4. Снапшот: створити накладну → змінити `NOM.price` → переекспортувати
   стару накладну → ціна стара (не змінилась).
5. Снапшот особи: змінити `PERS` рядок підрозділу → стара накладна показує
   стару особу.
6. Номер: дублікат номера → попередження; різні типи операції → різні
   префікси.
7. `amount_to_words_uk(Decimal("76453.20"))` → еталонний рядок (розділ 7.6).
8. Відкриття у Excel/LibreOffice — нуль помилок формул (#REF!/#VALUE!).
9. Кодування: жодної латинської `i/e/o` всередині українських слів
   прописом.

## 9. Заборони

- Не хардкодити обчислені суми — лише формули в I та підсумку.
- Не копіювати зіпсований пропис з оригіналу.
- Не змінювати ширини колонок/шрифт/орієнтацію/поля шаблону.
- Не писати в праву/нижню комірку merged-діапазону — лише ліва-верхня.
- Excel-експорт не звертається до поточних довідників — лише снапшот.
- Не покладатись сліпо на автокорекцію merge/формул openpyxl при
  insert/delete — перевіряти явно (за «якорями»).
- Блок фінансово-економічного органу не відновлювати — він видалений з
  шаблону навмисно.
